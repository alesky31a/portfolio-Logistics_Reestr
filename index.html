О проекте

Контекст и роль аналитика

Полный цикл SA · Статус: MVP реализован

Контекст: Логистическая компания «Bridging the Gap» (РБ). Два пользователя системы — Логист и Бухгалтер. Учёт рейсов ведётся вручную в MS Word, документы передаются физически на бумаге.

Роль аналитика: выявить критические боли процесса, спроектировать автоматизированное решение, задокументировать его от BRD до SQL DDL и покрыть тест-кейсами.

Единая архитектура: все артефакты (Data Dictionary, SQL DDL, тест-кейсы) описывают одну и ту же структуру, которую физически создаёт код функции initLogisticsSystem().

Артефакт — Анализ AS-IS

Три критические боли бизнеса

🔴 Ловушка «Copy-Paste» — риск недействительности ТТН Логист копирует строки прошлых рейсов, наследуя устаревшие госномера. По законодательству РБ единственная опечатка в госномере делает ТТН юридически недействительной → штрафы при налоговых проверках, риск признания сделок ничтожными.
🔴 Слепая зона по фактическому весу — заморозка оборотного капитала Данные о тоннаже приходят с задержкой. Бухгалтер принимает рейсы «вслепую» и не может вовремя выставить счёт реализации → кассовые разрывы, задержка оплат.
🔴 Ошибки в тарифных расчётах — прямые финансовые потери Две модели: «Фрахт» (фиксированная ставка) и «ТКМ» (Вес × Тариф). Ручные расчёты на калькуляторе и путаница ставок → ошибки при сверке с перевозчиками.

Бизнес-правила

BR_LOG_01 — Модель «Фрахт» (фиксированная ставка) Применяется для рейсов с договорной фиксированной стоимостью.
Итоговая стоимость = Ставка реализации
Вес и ставка покупки в расчёт не входят (ставка покупки фиксируется для сверки с перевозчиком).
BR_LOG_04 — Модель «ТКМ» (тоннажная) Применяется при перевозке сыпучих грузов (зерно, кукуруза).
Итоговая стоимость = Вес (Тн) × Ставка реализации
Если вес пуст или ≤ 0 → итоговая стоимость = 0.00, ячейка очищается до заполнения корректного веса.
BR_LOG_05 — Workflow Guard (защитный шлюз) Аппаратная блокировка некорректного перехода статуса с откатом транзакции.

Архитектура БД: SQL DDL

Версия 2.0 · Статус: Утверждено · PostgreSQL · Схема 1:1 с Apps Script MVP

Схема для миграции с Google Sheets на PostgreSQL. Структура таблиц и enum-значения полностью совпадают с тем, что создаёт initLogisticsSystem() в Google Apps Script.

🗄 PostgreSQL DDL — Справочники
-- СПРАВОЧНИК 1: ПЕРЕВОЗЧИКИ
CREATE TABLE dict_carriers (
    carrier_id    SERIAL PRIMARY KEY,
    carrier_name  VARCHAR(200) NOT NULL,
    unp           VARCHAR(9) UNIQUE NOT NULL
                  CHECK (unp SIMILAR TO '[0-9]{9}')
);

-- СПРАВОЧНИК 2: ТРАНСПОРТНЫЕ СРЕДСТВА
CREATE TABLE dict_vehicles (
    vehicle_id    SERIAL PRIMARY KEY,
    truck_plate   VARCHAR(20) UNIQUE NOT NULL,
    trailer_plate VARCHAR(20),
    fleet_type    VARCHAR(20) NOT NULL
                  CHECK (fleet_type IN ('Собственный', 'Привлеченный'))
);

-- СПРАВОЧНИК 3: ВОДИТЕЛИ
CREATE TABLE dict_drivers (
    driver_id   SERIAL PRIMARY KEY,
    driver_name VARCHAR(150) NOT NULL,
    phone       VARCHAR(30)

🗄 PostgreSQL DDL — Основная таблица с Workflow Guard constraint
CREATE TABLE operations_flights (
    id                SERIAL PRIMARY KEY,
    trip_date         DATE NOT NULL,
    logistician_name  VARCHAR(150) NOT NULL,
    accountant_name   VARCHAR(150),
    status            VARCHAR(50) NOT NULL DEFAULT 'Создан'
                      CHECK (status IN (
                          'Создан', 'В процессе',
                          'Рейс выполнен', 'Готов к учету', 'На доработке'
                      )),
    shipper_address   TEXT NOT NULL,
    consignee_address TEXT NOT NULL,
    cargo_type        VARCHAR(100) NOT NULL,

    -- ON DELETE RESTRICT: запрещаем удалять справочные записи
    carrier_id  INT NOT NULL REFERENCES dict_carriers(carrier_id) ON DELETE RESTRICT,
    truck_id    INT NOT NULL REFERENCES dict_vehicles(vehicle_id) ON DELETE RESTRICT,
    trailer_id  INT NOT NULL REFERENCES dict_vehicles(vehicle_id) ON DELETE RESTRICT,
    driver_id   INT NOT NULL REFERENCES dict_drivers(driver_id)   ON DELETE RESTRICT,

    weight_tn     DECIMAL(10,2),
    calc_model    VARCHAR(20) NOT NULL
                  CHECK (calc_model IN ('Фрахт', 'ТКМ')),
    purchase_rate DECIMAL(10,2) NOT NULL CHECK (purchase_rate >= 0),
    sales_rate    DECIMAL(10,2) NOT NULL CHECK (sales_rate >= 0),
    total_cost    DECIMAL(10,2) NOT NULL DEFAULT 0.00,
    fleet_type    VARCHAR(20) NOT NULL
                  CHECK (fleet_type IN ('Собственный', 'Привлеченный')),

    -- Аппаратный Workflow Guard на уровне СУБД
    CONSTRAINT chk_weight_for_tkm CHECK (
        NOT (
            status = 'Готов к учету'
            AND calc_model = 'ТКМ'
            AND (weight_tn IS NULL OR weight_tn <= 0)
        )
    )
);
🔁 Триггер авторасчёта — зеркало calculateTotal_()
CREATE OR REPLACE FUNCTION fn_calculate_total_cost()
RETURNS TRIGGER AS $$
BEGIN
    IF NEW.calc_model = 'Фрахт' THEN
        NEW.total_cost := NEW.sales_rate;

    ELSIF NEW.calc_model = 'ТКМ' THEN
        IF NEW.weight_tn IS NOT NULL AND NEW.weight_tn > 0 THEN
            NEW.total_cost := NEW.weight_tn * NEW.sales_rate;
        ELSE
            NEW.total_cost := 0.00;
        END IF;
    END IF;

    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_calculate_total_cost
BEFORE INSERT OR UPDATE ON operations_flights
FOR EACH ROW
EXECUTE FUNCTION fn_calculate_total_cost();
💡 CONSTRAINT chk_weight_for_tkm Гарантирует, что Workflow Guard работает не только в Apps Script, но и при прямых запросах к БД — например, со стороны будущего API или мобильного клиента.

2Меню Расширения → Apps Script
3Удалите стандартный код из Код.gs, вставьте финальный скрипт, сохраните (Ctrl+S)
4В выпадающем списке выберите initLogisticsSystem → нажмите ▶ Выполнить

⚙️ Apps Script — Финальный скрипт (основные блоки)
// КОНФИГУРАЦИЯ: централизованные константы
const SHEET_REGISTRY = "Реестр";
const SHEET_VEHICLES = "Справочник_ТС";
const SHEET_DRIVERS  = "Справочник_Водители";
const SHEET_CARRIERS = "Справочник_Перевозчики";

const COL = {
  STATUS: 5, CARRIER: 9, TRUCK: 10,
  WEIGHT: 13, MODEL: 14,
  RATE_PURCHASE: 15, RATE_SALE: 16, TOTAL_COST: 17
};

// БИЗНЕС-ЛОГИКА: чистые функции
function calculateTotal_(model, weight, rateSale) {
  if (model === "Фрахт") return rateSale;
  if (model === "ТКМ") {
    return (weight > 0 && rateSale > 0) ? weight * rateSale : 0;
  }
  return 0;
}

function isWorkflowBlocked_(newStatus, model, weight) {
  if (newStatus !== "Готов к учету") return false;
  return model === "ТКМ" && (weight === "" || weight === null || weight <= 0);
}

// ТРИГГЕР onEdit
function onEdit(e) {
  if (!e) return;
  const range = e.range;
  const sheet = range.getSheet();
  if (sheet.getName() !== SHEET_REGISTRY) return;
  const row = range.getRow();
  if (row === 1) return;
  const col = range.getColumn();
  const status   = sheet.getRange(row, COL.STATUS).getValue();
  const weight   = sheet.getRange(row, COL.WEIGHT).getValue();
  const model    = sheet.getRange(row, COL.MODEL).getValue();
  const rateSale = sheet.getRange(row, COL.RATE_SALE).getValue();

  if (col === COL.STATUS && isWorkflowBlocked_(status, model, weight)) {
    range.setValue(e.oldValue || "В процессе");
    SpreadsheetApp.getUi().alert(
      "🛑 Workflow Guard — Транзакция заблокирована!\n\n" +
      "Для модели ТКМ поле 'Вес (Тн)' обязательно и должно быть > 0."
    );
    return;
  }

  if ([COL.WEIGHT, COL.MODEL, COL.RATE_PURCHASE, COL.RATE_SALE].includes(col)) {
    const total = calculateTotal_(model, weight, rateSale);
    sheet.getRange(row, COL.TOTAL_COST).setValue(total > 0 ? total : "");
  }
}
